Architecture Overview

Pseudocode uses a hybrid execution model combining bytecode interpretation with trace-based JIT compilation. Source code is compiled to bytecode and executed by a stack-based virtual machine. Hot loops are detected and compiled to native x86-64 machine code.

Compilation Pipeline

The compiler is a single-pass design that directly emits bytecode while parsing. This approach trades some optimization opportunities for simplicity and fast compile times.

Lexer

The lexer (lexer.c) tokenizes source code into a stream of tokens. Key features:

  • Single-character lookahead
  • Line and column tracking for error messages
  • String interpolation support
  • Keyword recognition via hash lookup

Compiler

The compiler (compiler.c) implements a Pratt parser for expressions with recursive descent for statements. It emits bytecode directly without building an AST.

Constant Folding

Compile-time evaluation of constant expressions like 2 + 3.

Local Variables

Stack-allocated locals with lexical scoping.

Closures

Upvalues capture variables from enclosing scopes.

Loop Optimization

Specialized opcodes for range-based loops.

Bytecode Format

Pseudocode uses a variable-length bytecode format. Most instructions are 1-3 bytes.

Opcode Operands Description
OP_CONSTANT index (1-2 bytes) Push constant from pool
OP_ADD none Pop two, push sum
OP_GET_LOCAL slot (1 byte) Push local variable
OP_LOOP offset (2 bytes) Jump backward
OP_CALL argc (1 byte) Call function

Virtual Machine

The VM (vm.c) is a stack-based interpreter with a single evaluation stack. It uses computed gotos (GCC/Clang) or a switch statement for opcode dispatch.

// Simplified dispatch loop
while (running) {
    uint8_t instruction = READ_BYTE();
    switch (instruction) {
        case OP_ADD: {
            Value b = pop();
            Value a = pop();
            push(val_num(as_num(a) + as_num(b)));
            break;
        }
        // ...
    }
}

Call Stack

Function calls use a separate call frame stack. Each frame tracks:

  • Return address (instruction pointer)
  • Stack base pointer
  • Closure reference

Tracing JIT Compiler

The JIT (jit.c, trace_recorder.c, trace_codegen.c) compiles hot loops to native x86-64 code using a tracing approach.

How It Works

  1. Profiling: Loop back-edges increment a counter
  2. Recording: When hot (>100 iterations), record a trace
  3. IR Generation: Convert trace to intermediate representation
  4. Register Allocation: Linear scan allocation
  5. Code Generation: Emit x86-64 using DynASM

Trace Recording: The recorder follows the actual execution path, capturing guards for type checks and branches. This creates a linear trace optimized for the common case.

IR Operations

typedef enum {
    IR_CONST,      // Load constant
    IR_LOAD,       // Load from slot
    IR_STORE,      // Store to slot
    IR_ADD,        // Arithmetic
    IR_SUB,
    IR_MUL,
    IR_DIV,
    IR_LT,         // Comparison
    IR_LE,
    IR_GUARD,      // Type/value guard
    IR_CALL_INLINE // Inlined function call
} IROpcode;

Function Inlining

Small functions (≤20 IR instructions) are automatically inlined during JIT compilation. This eliminates call overhead for helper functions in hot loops.

Value Representation

Pseudocode uses NaN-boxing to represent all values in 64 bits. IEEE 754 doubles have a large space of NaN representations that can encode pointers and tagged values.

Type Representation
Number Raw IEEE 754 double
Nil Quiet NaN with tag 0x1
Boolean Quiet NaN with tag 0x2/0x3
Object Quiet NaN with 48-bit pointer 
// NaN-boxing implementation
#define QNAN     0x7ffc000000000000
#define TAG_NIL  1
#define TAG_FALSE 2
#define TAG_TRUE  3

#define IS_NUM(v)  (((v) & QNAN) != QNAN)
#define as_num(v)  val_to_num(v)
#define val_num(n) num_to_val(n)

Memory Management

Pseudocode uses a simple mark-and-sweep garbage collector. All heap objects are linked in a list for traversal during collection.

Object Types

  • OBJ_STRING: Immutable strings with cached hash
  • OBJ_ARRAY: Dynamic arrays with capacity doubling
  • OBJ_DICT: Hash tables with open addressing
  • OBJ_FUNCTION: Compiled function with bytecode chunk
  • OBJ_CLOSURE: Function with captured upvalues
  • OBJ_CLASS: Class with method table
  • OBJ_INSTANCE: Object instance with field table

GC Triggers

Collection is triggered when heap size exceeds a threshold (default 8MB). The threshold grows after each collection to amortize GC cost.
